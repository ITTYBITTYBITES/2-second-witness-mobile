# Witness Threads Future Framework

**Status:** Future framework only. Do not build Story Mode or full narrative infrastructure.
**Current decision:** document now; conditional prototype only after flagship retention/fairness/device gates are complete.

---

# 1. Purpose

Witness Threads are hidden relationships among separate, standalone Witness Moments. They may eventually make a returning player realize that an ordinary detail they saw before has appeared again in a new context.

The desired thought is:

> **“I wonder what I have already seen that I didn’t understand yet.”**

The prohibited thought is:

> **“I need to finish the story.”**

The act of noticing remains the product. A future connection is only a retrospective layer of meaning.

---

# 2. Core rules

1. **Scenes remain standalone.** Every scene is enjoyable, fair, scorable, and explainable with no thread knowledge.
2. **Connections are discovered.** They surface only after enough normal evidence exists; never as assigned tasks.
3. **No punishment for missing details.** Thread discovery cannot require perfect scores, exact memory, schedule adherence, or replay remediation.
4. **No campaign structure.** No chapters, episode order, Cases menu, quest list, named characters, completion bars, or “next story” call-to-action.
5. **Normal reveal comes first.** Current-scene evidence resolves before any connection cue.
6. **No FOMO.** No time limit, notification pressure, missed-day consequence, or hidden-content warning.
7. **Content is factual; interpretation stays open.** Product can state a detail recurred, not narrate a plot about why.

---

# 3. Connection structure

```text
Seed scene → unrelated normal play → Echo scene → unrelated normal play → Reframe scene
→ optional connected-evidence archive relation
```

| Parameter | Framework standard |
|---|---:|
| Minimum scenes | 3 |
| Preferred size | 3–4 scenes |
| Maximum size | 5 scenes |
| Primary evidence anchor | One recurring object/motif/relation per thread |
| Spacing | Prefer at least one unrelated normal moment between beats |
| Eligibility | Played/witnessed moments, never flawless score gate |
| Selection priority | Below accessibility, valid Brief state, fairness, freshness, and explicit player choice |

The first and second beats are silent. The third beat may surface a connection only after its normal evidence reveal is understood.

---

# 4. Connected evidence types

| Type | Example | Why it fits |
|---|---|---|
| Object journey | Folded map appears in office, travel, garden contexts. | Material trace across ordinary worlds. |
| Routine echo | Same practical object/arrangement appears in different daily tasks. | Implies continuity without named character/story. |
| Visual motif | Marked corner, binding, nonverbal symbol, material pattern recurs. | Supports visual hindsight without text/lore. |
| Spatial echo | Familiar object shifts relation to a new anchor/place. | Uses core scene-relationship grammar. |
| Evidence reframe | Incidental earlier detail later becomes a meaningful repeated reference. | Makes reveal feel deeper without invalidating original scene. |

Do not use secret codes, microscopic text, color-only callbacks, named possessions, crime clues, or props that imply a traditional detective plot.

---

# 5. Reveal and archive evolution

## Reveal

After normal scene truth settles:

1. current scene returns;
2. current target/relation is highlighted;
3. factual explanation appears;
4. only then, optional quiet callback shows prior related evidence;
5. one factual line states recurrence;
6. player may view archive relation or continue normal Brief.

Example:

> **This folded map appeared in two earlier moments you witnessed.**

No “case complete,” no count, no next-thread teaser, no plot exposition.

## Archive

The Witness Record may later organize discovered relations as **connected evidence**:

- original scene references remain in context;
- recurring detail is marked consistently;
- personal encounter order can be shown;
- no empty slots, locked cards, or completion percentage appear;
- archive relationship is optional and ignorable.

The archive should resemble memory becoming organized, not a detective board/corkboard/mission map.

---

# 6. Future additive metadata boundary

If and only if approved after validation, metadata may be added at Content/Game layer:

- `thread_id`;
- `thread_beat` (seed/echo/reframe);
- `connection_anchor_id`;
- scene-specific anchor/reveal geometry;
- factual connection-copy key;
- display eligibility/version.

This metadata must not:

- alter normal answer truth/scoring;
- create a second save/profile store;
- add family-specific branches to shared runtime/Home/Programs;
- override player-selected content;
- require network, account, notification, or new navigation routes.

---

# 7. Prototype gate

A one-thread prototype is authorized only when all are demonstrated:

- Scene Investigation creates voluntary replay/return behavior on its own.
- Evidence reveal is understood and emotionally satisfying after correct/missed outcomes.
- Players naturally remember or refer to prior ordinary scenes across sessions.
- Current Brief/Record systems are clear and non-pressuring.
- Device/accessibility/performance release gates are green.
- Content team can preserve standalone scene quality while carrying recurring-detail QA cost.

## Prototype scope

One three-scene material thread. Existing scene worlds. Existing runtime. Normal scene questions. One optional archive/reveal callback. No player-facing thread navigation or progress.

## Prototype success

Players say the connection feels like a surprise in hindsight and still describe the product primarily as an observation experience. If it creates confusion, task pressure, detective-story expectation, or content-quality regression, reject/defer it.
