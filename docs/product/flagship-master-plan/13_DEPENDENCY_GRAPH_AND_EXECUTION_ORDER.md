# Dependency Graph and Execution Order

**Purpose:** make update dependencies explicit so the flagship evolves in the correct order and working architecture is not destabilized by premature scope.

---

# 1. Core dependency principle

The flagship cannot be polished into existence from the outside. It must become clear in this order:

```text
First Witness Moment clarity
→ Evidence Reveal quality
→ Scene quality/fairness/replay pipeline
→ Return ritual / Witness Brief
→ Witness Record meaning
→ Premium presentation
→ Scalable content
→ Device excellence
→ Future connected evidence preparation
→ Flagship release
```

A downstream layer must never be used to conceal an upstream uncertainty.

# 3. Detailed prerequisites

| Update | Must be true before starting | Must be true before completing | Cannot be deferred to later update |
|---|---|---|---|
| **1 Foundation** | Protected baseline/first-session policy documented. | New player understands Witness Moment; first scene is fair; tutorial path coherent. | First-session clarity and runtime launch authority. |
| **2 Reveal** | Current result truth/evidence data audited; scene target/reveal semantics known. | Context-first reveal works across question types/settings/devices. | Correct/missed fairness and evidence hierarchy. |
| **3 Scene Pipeline** | Reveal tells content team what evidence it must provide. | Each production scene/template has visual/fairness/replay standard and seed/device review. | Quality gates for scene content. |
| **4 Brief** | First/normal scenes and result continuation are clear. | One truthful current-brief/resume semantics is understandable. | No-FOMO/one-primary-action product definition. |
| **5 Record** | Brief state and moment result data have meaning. | Players understand private record without dashboard/pressure. | Save/profile migration safety. |
| **6 Presentation** | Information hierarchy is validated, not speculative. | Motion/audio/art make validated loop premium and accessible. | Reduced Motion/audio-off/high-contrast equivalence. |
| **7 Content** | Scene quality pipeline and art workflow are proven. | New content remains fair, distinct, and performant. | Per-template review/validation discipline. |
| **8 Device** | Stable representative flagship content/presentation exists. | Physical device/accessibility/performance/signed artifact gates pass. | Timing fairness, save safety, Android reliability. |
| **9 Threads Prep** | Core retention/reveal/record/device evidence exists. | Only additive content/archive readiness exists; no campaign layer. | Standalone-scene and no-obligation rules. |
| **10 Completion** | Updates 1–8 hard gates pass; Update 9 remains bounded. | Store, first session, retention, content, device, and release evidence align. | No late feature scope. |

# 4. Critical path

The critical path is:

```text
U1 → U2 → U3 → U4 → U5 → U6 → U8 → U10
```

- **U7 Content Expansion** can begin only after U3 standards are proven, but must not delay U8 device validation with unbounded asset scope.
- **U9 Threads Preparation** is explicitly non-critical and may be skipped entirely for flagship release. It cannot block U10.
- Store assets/evolution work can begin conceptually after U2/U3, but final capture and claims depend on U6/U8/U10.

# 5. Parallel work allowed

## Safe parallel documentation/research work

While implementation of one update is underway, teams may prepare—but not build—later work:

| Active update | Safe parallel preparation |
|---|---|
| U1 | Evidence reveal content/copy matrix; first-session research protocol. |
| U2 | Scene quality rubric; asset pipeline briefs; reveal device capture plan. |
| U3 | Brief semantics research; scene content capacity planning. |
| U4 | Witness Record information architecture research; progression interpretation studies. |
| U5 | Presentation mood/audio/art exploration using validated scene/result hierarchy. |
| U6 | Content template proposals and device matrix scheduling. |
| U7 | Store screenshot/storyboard planning from real product behavior. |
| U8 | Witness Threads content documentation only. |

## Work that must not run in parallel

- Full Home redesign while U4 brief semantics are unresolved.
- Large scene expansion while U3 fairness/art pipeline is unproven.
- Narrative/Threads implementation while U1–U5 core retention is unproven.
- Store/trailer claims that depict unshipped future behavior.
- Architecture replacement while protected runtime/saves/navigation meet current needs.

# 6. Decision gates

## Gate A — Is the Witness Moment real?

After U1/U2: first-time players understand the loop, fair misses work, and reveal creates voluntary continuation.

**If no:** stop roadmap expansion; repair U1/U2.

## Gate B — Are scenes sustainably good?

After U3: visual/fairness/replay process produces meaningful scenes across seeds/tier/device/human sessions.

**If no:** do not add content worlds, Brief complexity, or premium storytelling.

## Gate C — Is there a return ritual?

After U4/U5: players understand a current Brief and value a record without pressure.

**If no:** simplify recommendation/progression rather than adding retention features.

## Gate D — Does premium survive reality?

After U6/U8: physical Android/accessibility/performance/signed artifact behavior meets flagship standard.

**If no:** block release; fix validated issues only.

## Gate E — Is there appetite for deeper hindsight?

After U9 research gate: players remember scenes and find a connection curious rather than obligating.

**If no:** leave Threads documented only.

---

# 7. Execution order rule

> A later update may enrich a proven layer. It may not replace the evidence missing from an earlier one.

This rule protects the app from becoming a polished collection of systems without one exceptional Witness Moment at its center.
