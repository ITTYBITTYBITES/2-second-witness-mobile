# Two Second Witness — Production UX/UI/Visual Audit

**Audit date:** 2026-07-10  
**Target:** Android Development preset (`Android_Development`, output `2sw-dev.apk`)  
**Baseline audited:** commit `84672f0edfcb1a794f46f011341f9a1eccacc731`  
**Orientation:** portrait-first  
**Method:** full scene/resource review, navigation/state tracing, asset inspection, Android export configuration review, contrast/touch-target analysis, and post-fix automated Godot/Android build checks.

> This table records the baseline issues found **before implementation**. “Resolution” is updated after remediation. Release-blocking means the issue should block a Google Play production rollout, not necessarily creation of an internal development APK.

## Coverage

| Area | Scenes / nodes audited |
|---|---|
| Launch | Native Godot boot splash; `PublisherSplashScreen`; `TitleSplashScreen`; title loading/privacy overlay |
| First launch | `TitleSplashScreen/PrivacyDialogLayer`; `PrivacyTermsDialog` and persistence path |
| Main app | `HomeScreen`; `ExperiencesScreen`; `ProfileScreen`; `SettingsScreen`; `AboutScreen`; `PlaceholderScreen` |
| Gameplay | `TutorialScreen`; `ObservationChallengeScreen`; `MemoryQuestionScreen`; `ResultScreen`; all five challenge assets/data records |
| Shell/overlays | `AppShell`; `TopBar`; `MainNavigation`; `LoadingOverlay`; `ErrorBanner` |
| Shared UI/assets | `AppButton`; `AppCard`; `ExperienceCard`; `SectionHeader`; launcher/adaptive icons; splash, menu, and gameplay imagery |
| Responsive/accessibility | Typography tokens, light/dark/high-contrast behavior, font scaling, reduced motion, safe areas, Android back, 48 dp targets |

## Findings

| ID | Severity | Scene and node location | Root cause | Recommended fix | Blocks Google Play? | Resolution |
|---|---|---|---|---|---|---|
| UX-01 | **Critical** | Project-wide; `project.godot [display]` and every `Control` using 14–60 unit sizing | The UI used a 1080×1920 logical canvas while treating pixels as density-independent units. A 48-unit target can render near 16 dp on a xxhdpi 1080 px phone; typography and margins are correspondingly undersized. | Use a 360×640 logical portrait canvas so one UI unit behaves like a dp at the reference size; retain responsive `canvas_items` expansion. | **Yes** — minimum touch target/readability failure | Fixed |
| UX-02 | **Critical** | `AboutScreen/Background`, gameplay backgrounds, tutorial and result backgrounds under light theme | Multiple scenes hard-coded a dark `ColorRect`, while ThemeService switched text to dark light-theme colors. This creates dark-on-dark unreadable screens. | Theme all screen backgrounds from tokens and refresh cached screens when the theme changes. | **Yes** — core screens become unreadable | Fixed |
| UX-03 | **Critical** | `assets/brand/android/icon_foreground.png`; legacy launcher icon | The adaptive foreground had a baked checkerboard and no alpha channel; the legacy icon also had a pre-rounded tile/white corners. | Regenerate a true transparent adaptive foreground and composite a full-bleed legacy icon from the canonical background/mark. | **Yes** — visibly broken launcher identity | Fixed |
| UX-04 | **High** | `TitleSplashScreen/PrivacyDialogLayer/PrivacyTermsDialog` | The node named `Scroll` was a `MarginContainer`, content height was unconstrained, action buttons were inside the scrolling/content column, and the 360-unit minimum panel width could overflow narrow phones. | Use an actual `ScrollContainer`, pin actions below it, size the dialog from the live viewport/safe area, and keep both actions at least 48 dp. | **Yes** — required first-launch action can clip | Fixed |
| UX-05 | **High** | Privacy links in `TitleSplashScreen`, `AboutScreen`, `ConfigService` | The in-app URL returned a 404 (`/two-second-witness/privacy`) while the existing Play policy is `/privacy-policy/`. | Centralize and use the live policy URL. Surface link-open failure to the app error UI. | **Yes** — broken required policy access | Fixed in app; hosted policy content still needs owner review |
| UX-06 | **High** | `PrivacyTermsDialog/Title`, footer, and primary button | UI said “Privacy & Terms” and “acknowledge these terms” but did not provide Terms of Service content or a terms link. “Accept” also implied consent beyond the disclosed local-storage notice. | Use precise “Privacy & Data” language and a “Continue” action unless a real Terms document is supplied. | **Yes** — misleading consent language | Fixed |
| UX-07 | **High** | `AppBoot`; `AccessibilityService`; `AnalyticsService` | Accessibility and analytics services were autoloaded but never initialized by boot. Font scale, reduced motion, high contrast, haptics preference, and analytics opt-out could remain at hard-coded runtime defaults. | Initialize both services after settings, and ignore analytics events before analytics initialization. | **Yes** — saved accessibility/privacy controls unreliable | Fixed |
| UX-08 | **High** | `SettingsScreen` slider rows | Every slider value change saved settings and triggered a deferred full screen rebuild, deleting the slider during a drag. | Keep live labels/audio updates local; persist on drag completion and avoid rebuilding the whole settings tree for ordinary value changes. | No (but severe usability defect) | Fixed |
| UX-09 | **High** | `ThemeService`, `SettingsScreen/Font Scale`, `High Contrast` | Font scale was displayed but not incorporated into theme sizes; high contrast only changed an internal boolean and had no visual effect. | Scale all token typography through ThemeService and apply a real high-contrast palette/border variant. | **Yes** — advertised accessibility controls were nonfunctional | Fixed |
| UX-10 | **High** | `TutorialScreen` `_ready`; `Margin/StepContainer` | The script referenced a nonexistent node (`Margin/StepContainer` instead of `Margin/VBox/StepContainer`), causing a runtime node lookup error when the route is opened. | Correct the path and make the tutorial independently loadable even though it is not in the current first-launch path. | No for current launch; yes if route is enabled | Fixed |
| UX-11 | **High** | `MainNavigation._on_tab_pressed` and `AppShell._on_nav_tab_selected` | Both the navigation component and shell navigated for one tab tap, producing duplicate route changes, screen refreshes, and analytics events. | Keep navigation ownership in AppShell; the tab bar should emit one intent only. | No | Fixed |
| UX-12 | **High** | Shell/root input | No Android/system-back handler routed `ui_cancel` through NavigationService. Device back could quit or behave inconsistently instead of returning from About/gameplay. | Handle `ui_cancel`, block it during mandatory first-launch privacy, and navigate back/home consistently. | **Yes** — Android navigation convention | Fixed |
| UX-13 | **High** | `MemoryQuestionScreen/OptionsContainer` feedback state | Correct green (`#2EE6A6`) with white text and error red with white text fail normal-text contrast; feedback relied strongly on color. Disabled-state styling could further dim it. | Use dark foreground on bright feedback fills and add explicit check/cross text markers. | **Yes** — answer feedback accessibility | Fixed |
| UX-14 | **High** | Shell content offsets plus tab screen `Margin` nodes | Shell reserved top/bottom chrome, while Home/Play/Profile/Settings also added 80/90-unit chrome margins. Content and the primary action were pushed down or below the fold on a true mobile logical canvas. | Let AppShell own safe/chrome insets; use 16-unit content margins in screens. Hide redundant top chrome on tab roots. | No | Fixed |
| UX-15 | **Medium** | `TopBar` on Home/Play/Profile/Settings | A web/desktop-like global bar duplicated page names plus Settings/Profile actions already present in bottom navigation; action text could crowd the app title at 360 dp. | Hide the top bar for root tabs; reserve it for back-navigation/gameplay context. | No | Fixed |
| UX-16 | **Medium** | `HomeScreen/VBox` | Level/XP/streak appeared before the primary “Play” action, increasing decision time and risking below-fold placement. | Put the single primary action directly below the hero; keep stats secondary. | No | Fixed |
| UX-17 | **Medium** | `ObservationChallengeScreen/ImageContainer/ObservationImage`; `_ready` | A 400-unit image minimum over-constrained short phones after app-bar insets. `_ready` also started the default challenge before `on_navigated_to` started the requested challenge, causing duplicate start/haptic behavior. | Use a flexible 240-unit minimum and start only after navigation data is applied. | No | Fixed |
| UX-18 | **Medium** | `ExperienceCard/PlayButton` and card typography | Play button had no guaranteed 48 dp height or coherent primary styling; card text used hard-coded 11/13 px sizes below the design scale. | Apply theme typography, 48 dp target, consistent button variants, wrapping, and spacing. | No | Fixed |
| UX-19 | **Medium** | `SettingsScreen` content | Production settings exposed silent audio controls (no audio assets), duplicate reduced-motion controls, no-op gameplay/screen-reader/crash controls, package ID, engine, and build internals. This reads like a debug/web settings panel. | Show only controls that currently work: theme, text size, reduced motion, contrast, haptics, local diagnostics, About, and reset. | No | Fixed |
| UX-20 | **Medium** | `SettingsScreen` toggles/sliders | Switches and slider tracks did not explicitly guarantee 48 dp interactive regions. | Give `CheckButton` and `HSlider` controls 48×48 minimum interaction size. | No | Fixed |
| UX-21 | **Medium** | `AboutScreen` | Version was stale (`2.0.0` versus export/config `4.0.0`), body text was mostly unthemed, package ID was user-facing, privacy URL was broken, and the website target was unavailable. | Bind version/URLs to ConfigService, style all text/buttons, use the live Play-listed website, and remove internal package data from the visible page. | No | Fixed |
| UX-22 | **Medium** | `ResultScreen` actions | Three equal default buttons did not communicate a primary next action; status icon used placeholder text (“OK”/“X”). | Make Next/Play Again primary, Replay secondary, Main Menu ghost, and pair iconography with textual status. | No | Fixed |
| UX-23 | **Medium** | `LoadingOverlay`, `ErrorBanner`, splash loading state | Loading animation ignored reduced motion and used a rotating fallback glyph; banner placement did not account for top cutouts; title splash offered no loading status during watchdog waits. | Use a non-rotating text status under reduced motion, safe-area error placement, and visible title loading copy. | No | Fixed |
| UX-24 | **Medium** | `ProfileScreen` | Raw anonymous ID and engine-like stat abbreviations add technical density; progress used hard-coded 12 px labels. | Explain that the profile is local/no-sign-in and use tokenized readable labels. | No | Fixed |
| UX-25 | **Medium** | Gameplay asset set `observation_challenge_01..05.png` | Challenge 1 is detailed illustrative art while challenges 2–5 are sparse flat vectors; fidelity and visual language change abruptly between rounds. | Re-art all challenge scenes as one cohesive set while preserving answer-critical object counts/colors/positions. | No | Open — content art direction task, not changed during architecture-preserving UI pass |
| UX-26 | **Low** | Publisher/title splash baked typography | Branding and subtitles are baked into images, so they do not scale with app text settings and are not independently exposed as semantic text. | Keep safe-zone crops now; replace baked text with accessible live labels in a future asset pass. | No | Open |
| UX-27 | **Low** | `TutorialScreen` icons | Emoji depended on OS fallback font and did not match the otherwise editorial visual system. | Use text-based Observe/Remember/Recall markers or bundled vector icons. | No | Fixed |
| UX-28 | **Low** | Challenge 1 question/detail copy | The prompt said “colored pencils,” while its explanation said “writing tools”; the visible set contains mixed writing instruments. | Use one precise noun in question and explanation. | No | Fixed |
| UX-29 | **High (external)** | Hosted privacy policy and Play listing | The hosted policy/listing describe AdMob, ads and IAP, while this APK states no advertising is currently included and ships those systems inactive. The Play listing also describes a materially different 80-scenario product. | Before production rollout, update the hosted policy, Data Safety/ads declarations, screenshots, and listing copy to exactly match this APK. | **Yes** | Open — Play Console/hosted legal content owner action |
| UX-30 | **Medium** | Gameplay/question/result layouts under large text | Question and result roots were fixed VBoxes with no vertical scroll fallback, risking clipping with maximum text scale or longer content. | Add ScrollContainers while retaining bottom-priority actions and 48 dp targets. | No | Fixed |

## Baseline release decision

**NO-GO for Google Play production** at the audited baseline because UX-01/02/03/04/05/06/07/09/12/13 and external UX-29 are release-blocking. The architecture and gameplay loop are viable, but the baseline did not meet production mobile density, first-launch policy access, theme/accessibility, icon, or Android navigation expectations.

## Post-fix acceptance criteria

- Godot 4.6.3 project imports with no script/scene errors.
- Automated route smoke test instantiates every screen/dialog at 360×640 and verifies critical nodes/touch targets.
- Android `Android_Development` export succeeds to `app/build/android/2sw-dev.apk`.
- APK manifest/package/version match `com.ittybittybites.the2secondwitness`, version `4.0.0` / code `40000`.
- APK payload contains modified scenes/scripts and corrected launcher assets.
- A physical-device pass is still required for OEM cutout/gesture behavior, TalkBack, haptics, external browser return, and final visual sign-off.
